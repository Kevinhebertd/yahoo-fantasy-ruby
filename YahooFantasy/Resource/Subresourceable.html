I'm starting to think that the SubResource is the way to go - and that each sub resource should be defined with the name, verb, request proc, parse proc, etc. This seems more logical with no dependency leakage.

Provides standard mixin functionality for subresources. All resources and collections within the Yahoo Fantasy API space allows for zero or more subresources.

The only benefit I see from pulling this out of the Resource::Base is file size. The fact that any `included Subresource` class needs to either extend Resource::Base or have it's own `api` method makes this a little confusing in the grand scheme of things.

Eventually this module should solely be used for:

  • maintaing the list of subresources

  • generating the attr_accessor and accessor! methods

Examples:

class SubresourcedClass
  include Subresourceable

  subresource :sub_entities, verb: :Get,
                             endpoint: '/subentities',
                             filters: [:name],
                             parser: ->(fc) { fc.resource.subentities }
end
